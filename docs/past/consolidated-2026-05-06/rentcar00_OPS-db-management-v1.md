# rentcar00_OPS DB 관리 구조 v1

## 1. 결론
`rentcar00_OPS` 는 **새 Supabase 프로젝트를 따로 만들지 않는다.**

기존 Supabase 프로젝트 `rentcar00-booking-system` 안에서,
기존 예약/차량/동기화 자산은 유지하고,
그 위에 **ops 업무처리 전용 계층**을 추가하는 방식으로 관리한다.

즉,
- 프로젝트는 유지
- 기존 테이블은 역할 유지
- ops 앱은 별도 논리 계층 추가
- Google Sheets 는 원본 유지, 최종 최소 상태 체크만 반영
로 간다.

---

## 2. 왜 새 DB를 만들지 않는가
### 2-1. 이미 재사용 가능한 자산이 있다
현재 프로젝트에는 아래 자산이 이미 있다.
- `cars`
- `ims_sync_reservations`
- `reservation_sync_runs`
- `reservation_sync_errors`
- `reservation_mappings` 개념

### 2-2. 새 DB를 만들면 관리 포인트가 늘어난다
새 DB 분리 시 아래를 별도로 이중 관리하게 된다.
- 차량 마스터
- 예약 원본
- 외부 예약 키 매핑
- sync 상태
- 운영 로그

이 앱은 독립 서비스라기보다
**기존 예약/차량 생태계 위에 업무처리 계층을 덧붙이는 앱**이라,
지금 시점에는 분리 비용이 더 크다.

### 2-3. 기존 DB의 성격과 앱 목적이 연결된다
- 기존 DB는 차량과 예약 외부 연동 흔적을 이미 가지고 있다.
- 새 앱은 그 위에서 상태 버튼과 로그를 추가로 운영하면 된다.
- 완전히 새로운 도메인을 만드는 것이 아니다.

---

## 3. 현재 기준 source of truth 정리
### 3-1. Google Sheets `차량현황 > 예약` 탭
현재 실무 원장에 가장 가깝다.

확인된 실제 컬럼:
- 예약ID
- 예약번호
- 차량번호
- 차종
- 대여일
- 반납일
- 배반차위치
- 임차인
- 고객번호
- 생년월일
- 소개처
- 결제금액
- 예약상태

현재 상태값 분포:
- 반납완료 33
- 예약중 16
- 예약취소 12
- 배차중 11

해석:
- 실무 원장으로는 충분하다.
- 하지만 업무처리 세부 상태는 없다.

### 3-2. `일정` 탭
`일정` 탭은 예약과 별도이지만, 실제 업무 진행에서는 매우 중요하다.

현재 확인된 컬럼:
- 일정번호
- 예약번호
- 차량번호
- Status
- Date
- 차종
- 위치
- 상세정보
- 가반납
- 예약ID
- 일정완료

해석:
- `예약ID` 로 예약 탭과 연결되는 일정 행이 존재한다.
- `Status` 는 배차/반납/기타 같은 일정 종류 역할을 한다.
- 다만 현재 탭에는 **레거시 메모형 행과 예약 연결형 행이 섞여 있다.**
- 따라서 일정 탭은 그대로 UI 원장으로 쓰기보다, 정규화 과정이 필요하다.

### 3-3. 이번 프로젝트에서 사용하지 않을 기존 Supabase 예약 계층
이번 ops 앱 기준으로는 아래를 기준 원장으로 사용하지 않는다.
- `ims_sync_reservations`
- `booking_orders`

이유:
- 사장님 기준 source of truth 가 Google Sheets `예약` + `일정` 탭으로 명확히 정해졌기 때문이다.
- 기존 Supabase 예약 테이블은 이번 앱의 업무 기준을 대표하지 않는다.

## 4. 추천 관리 구조
같은 Supabase 프로젝트 안에서 아래 4층으로 관리한다.

### A. 차량 마스터 층
현재 유지 대상:
- `cars`

역할:
- 차량 기준 정보
- 차량번호/차명/연식/옵션
- 외부 차량 키 보존

### B. 원본 예약 층
현재/예정 source:
- Google Sheets `예약` 탭
- Google Sheets `일정` 탭

역할:
- 외부/기존 원장의 값 보존
- 원본 상태 추적
- sync/비교 기준

원칙:
- 원본 예약층은 사람이 누른 업무 버튼 상태를 직접 소유하지 않는다.

### C. ops 업무처리 층
새로 필요:
- 업무앱 메인 projection
- 업무 상태 묶음
- 수동 상태 보정
- 확인 필요 플래그
- 메모

역할:
- 앱 탭 분류 기준
- 상태별 버튼 처리 결과 저장
- `오늘배차`, `반납일`, `확인 필요` 같은 앱 상태 계산

원칙:
- 이 층이 Flutter 앱의 사실상 메인 읽기/쓰기 대상이 된다.

### D. 로그/템플릿/연동 층
새로 필요:
- 업무 액션 로그
- 문자 템플릿
- 외부 연결 상태
- 연락처 저장/발송/공유 기록
- 시트 최종 상태 반영 기록

역할:
- 감사 추적
- 재발송 이력
- 외부 서비스 붙일 준비
- 어떤 시점에 시트 상태 체크를 반영했는지 추적

## 6. 키 관리 원칙
### 6-1. `예약ID`를 원본 기준 키 후보 1순위로 본다
확인 결과:
- `예약ID` 는 72/72 채워짐
- `예약번호` 는 71/72만 채워짐

따라서 현재 기준:
- `예약ID` = 원본 행의 안정 키 후보
- `예약번호` = 업무상 노출/외부 식별 보조 키 후보

단,
최종 확정 전 아래를 더 확인해야 한다.
- AppSheet가 어떤 키를 실제 row key로 쓰는지
- `예약ID` 생성 규칙이 완전히 안정적인지

### 6-2. 차량 연결은 `차량번호` 우선
시트에는 `차량번호`가 있고,
기존 Supabase `cars` 에도 `car_number` 가 있다.

현재 가장 현실적인 연결 기준:
- 1차: 차량번호 match
- 2차: 필요 시 차종/보조 메타로 보정

## 7. 앱 상태는 원본 상태와 분리한다
시트 원본 상태:
- 예약중
- 배차중
- 반납완료
- 예약취소

앱 상태:
- 예약중
- 오늘배차
- 배차중
- 반납일
- 완료/기록

원칙:
- 원본 `예약상태` 는 그대로 보존
- 앱 탭 상태는 별도 계산 상태로 둔다

이유:
- `오늘배차`, `반납일` 은 날짜 + 처리 여부 기반이다.
- 원본 시트 컬럼 하나로는 표현되지 않는다.

## 8. 운영 관리 원칙
### 8-1. 원본을 덮어쓰지 않는다
- Google Sheets 원본은 원본대로 둔다.
- 앱 전용 상태는 Supabase 내부 상태 계층에 저장한다.
- 시트에는 최종적으로 필요한 최소 상태 체크만 반영한다.

### 8-2. 수동 fallback을 허용한다
- IMS / 문자 / 탁송 / 계약 연동이 미완성이어도
- 사람이 체크 버튼과 메모로 계속 운영 가능해야 한다.

### 8-3. 중요한 액션은 전부 로그화한다
- 문자 생성
- 요청문 생성
- 상태 변경
- 반납 완료
- 인수 완료
- 수동 보정

### 8-4. 원본 sync와 앱 상태 변경을 분리한다
- sync는 원본 feed 갱신
- 앱 상태는 ops 처리 결과 갱신
- 시트 상태 반영은 별도 outbound 반영 단계로 분리한다

셋은 같은 시점에 일어날 수 있어도,
소유권은 다르게 본다.

  - 탭 조회 기준
  - 상세 조회 기준
  - 버튼 실행 mutation 기준

## 10. 현 시점 최종 판단
사장님 기준 현재 가장 안정적인 방향은 아래다.

1. **새 Supabase DB는 만들지 않는다**
2. **기존 `rentcar00-booking-system` 프로젝트를 유지한다**
3. **Google Sheets `예약` + `일정` 탭만 원본 기준으로 본다**
4. **같은 프로젝트 안에 ops 전용 상태/로그/템플릿 계층을 추가한다**
5. **기존 `ims_sync_reservations`, `booking_orders` 는 이번 앱 원장 기준에서 제외한다**
6. **최종적으로는 시트의 상태 체크만 최소 반영한다**

이 구조가 지금 가장 단순하고, 확장 여지도 좋다.
