# rentcar00_OPS Master Spec

## 1. 목적
이 앱은 예약 현황을 보기 위한 일반 앱이 아니라,
**예약 상태별 탭 리스트에서 실제 업무를 처리하는 현황판 앱**이다.

원본은 Google Sheets `차량현황` 문서의 아래 2개 탭이다.
- `예약`
- `일정`

앱은 원본 시트를 직접 업무 DB처럼 쓰지 않고,
업무 상태와 로그는 Supabase에서 관리한다.
최종 완료 시 필요한 상태 체크만 시트에 반영한다.

---

## 2. 고정 기준
- 메인 구조는 **예약 상태별 탭 리스트**다.
- 상세 화면은 **보조**다.
- `ims_sync_reservations`, `booking_orders` 는 이번 앱 기준에서 사용하지 않는다.
- 원본 데이터는 Google Sheets `예약` + `일정` 탭이다.
- 업무 상태/로그는 Supabase에서 관리한다.
- 최종적으로 시트에는 필요한 상태 체크만 반영한다.

---

## 3. 메인 탭 구조
확정된 메인 탭은 아래 5개다.
- 예약중
- 오늘배차
- 배차중
- 반납일
- 완료

이 앱의 메인은 이 5개 탭이며,
사용자는 각 탭 리스트에서 카드를 보고 바로 업무를 처리해야 한다.

---

## 4. 탭별 목적
### 4-1. 예약중
아직 배차 전 단계의 예약을 관리하는 탭.
사전 확인, 요청, 준비 업무를 처리한다.

### 4-2. 오늘배차
오늘 배차가 필요한 예약을 관리하는 탭.
배차 직전/당일 실행 업무를 처리한다.

### 4-3. 배차중
차량이 실제 이용 중인 예약을 관리하는 탭.
운행 중 확인, 연장, 반납 전 안내 성격의 업무를 처리한다.

### 4-4. 반납일
반납이 예정되었거나 반납 완료 처리가 필요한 예약을 관리하는 탭.
반납 관련 액션과 완료 처리를 담당한다.

### 4-5. 완료
완료된 예약 기록을 조회하는 탭.
검색/이력 확인 용도다.

---

## 5. 탭별 카드 원칙
각 탭의 카드는 아래 원칙을 따른다.
- 한눈에 상태를 판단할 수 있어야 한다.
- 지금 바로 눌러야 하는 버튼이 보여야 한다.
- 카드만 보고 다음 액션을 판단할 수 있어야 한다.
- 상세 화면에 들어가지 않아도 주요 처리가 가능해야 한다.

카드 공통 필드 후보:
- 예약번호
- 예약자명
- 차량번호
- 차종
- 대여일시
- 반납일시
- 위치 요약
- 현재 업무상태
- 확인 필요 여부

세부 카드 필드는 탭별 기능 리스트에서 확정한다.

---

## 6. 확정된 업무 기능 후보
사장님이 직접 말한 기능 기준으로 아래를 우선 후보로 둔다.
- 연락처 저장
- 신분증 요청
- 주소 요청
- 확보 확인
- 배차 안내
- 탁송 요청
- 기사 배정/안내
- 계약서 확인
- 반납 전 안내
- 연장 확인
- 반납 안내
- 반납 완료

주의:
- 위 기능은 **후보 목록**이다.
- 어떤 기능이 어느 탭에서 노출되는지는 다음 문서에서 탭별로 확정한다.

---

## 7. 상세 화면 역할
상세 화면은 메인이 아니다.
역할은 아래로 제한한다.
- 카드에서 다 못 담는 원본 정보 확인
- 일정/메모/로그 확인
- 예외 상황 처리
- 보조 액션 실행

즉,
주요 업무 흐름은 탭 리스트와 카드 액션 중심으로 설계한다.

---

## 8. 이번 문서에서 아직 확정하지 않은 것
아래는 아직 확정하지 않는다.
- 각 탭의 진입 조건 세부 규칙
- 각 탭 카드의 최종 표시 필드
- 각 버튼의 정확한 노출 탭
- 상태 전이 규칙 세부값
- 완료 시 시트에 반영할 정확한 체크 항목

이 항목들은 추측하지 않고 다음 단계에서 확정한다.
