## 3. 시트 기준으로 봤을 때 현재 DB/앱에 추가로 필요한 부분
업무처리 앱 기준으로 아래가 비어 있다.

- Google Sheets에는 없는 업무앱 전용 상태
- 연락처 저장 여부
- 신분증 요청/확보 상태
- 주소 요청/확보 상태
- 계약서 발송/서명/내부확인 상태
- 고객 배차안내 발송 상태
- 탁송 요청 / 기사 배정 / 기사 안내 상태
- 고객 인수 완료 처리
- 반납 전 안내 / 연장 확인 상태
- 반납 안내 / 회수 처리 / 반납 완료 상태
- 업무 버튼 실행 로그
- 문자 템플릿 관리
- 수동 상태 보정 및 자동 분류 충돌 표시

즉, 현재 DB만으로는 `조회`는 일부 가능하지만 `업무 처리`까지는 부족하다.

## 4. 시트 + DB를 함께 본 1차 해석
### 해석 A
`booking_orders`를 메인으로 쓰는 방식은 현재 범위상 맞지 않다.
- 데이터가 거의 없음
- 웹 예약 주문 중심
- 운영 앱 업무 상태를 담기에 구조가 다름

### 해석 B
`Google Sheets 예약 탭`을 최우선 기준 원장으로 보고,
필요 시 `ims_sync_reservations`를 보조 feed 또는 비교 기준으로 사용하는 방식이 더 적합하다.
그 위에 업무처리용 상태/로그/템플릿 계층을 추가하는 구조가 맞다.

### 해석 C
실제 시트 컬럼을 확인한 결과,
현재 원장만으로는 앱이 필요로 하는 업무 상태를 충분히 표현하지 못한다.
즉, `원본 시트 컬럼`과 `업무앱 내부 상태 컬럼`을 분리해야 한다.

다만 기존 `booking_orders`를 그대로 업무앱 메인으로 쓰기는 부적합하다.

실제 시트 컬럼을 본 기준 현재 가장 현실적인 방향:

4. `오늘배차`, `반납일` 같은 앱 탭은 날짜/업무상태 기반 계산으로 생성
