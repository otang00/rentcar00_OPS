# rentcar00_OPS 개념 스키마 v1

## 1. 문서 목적
이 문서는 `rentcar00_OPS` 의 DB 설계를 시작하기 위한 **개념 스키마** 문서다.

중요:
- 실제 물리 테이블명/컬럼명은 아직 확정하지 않는다.
- 이 문서는 엔티티와 필드 묶음, 소유권, 관계를 먼저 잠그기 위한 것이다.

---

## 2. 전체 엔티티 구성
현재 구조는 아래 7개 엔티티 묶음으로 보는 것이 가장 적절하다.

1. 차량 마스터
2. 원본 예약 feed
3. 원본 일정 feed
4. ops 예약 projection
5. ops 상태 세부 정보
6. 업무 액션 로그
7. 문자 템플릿 / 외부 연결 메타
8. 시트 상태 반영 메타

---

## 4. 엔티티 관계
### 관계 1
차량 마스터 1 ↔ N ops 예약 projection

### 관계 2
원본 예약 feed 1 ↔ 1 또는 N ops 예약 projection
- 일반적으로 1:1 지향
- 예외 상황 보정 가능성은 열어둔다

### 관계 3
원본 일정 feed N ↔ 1 ops 예약 projection
- 예약 1건에 여러 일정 행이 연결될 수 있다

### 관계 4
ops 예약 projection 1 ↔ 1 ops 상태 세부 정보

### 관계 5
ops 예약 projection 1 ↔ N 업무 액션 로그

### 관계 6
ops 예약 projection 1 ↔ N 외부 연결 메타

### 관계 7
문자 템플릿 1 ↔ N 로그(간접)
- 어떤 템플릿이 사용됐는지 기록 가능

### 관계 8
ops 예약 projection 1 ↔ N 시트 상태 반영 메타
- 어떤 최종 상태를 시트에 반영했는지 추적

## 5. 상태 계산 소유권
### 원본 예약 feed가 소유하는 것
- 원본 예약 상태 원문
- 원본 일정
- 원본 고객/차량/위치 텍스트

### ops projection이 소유하는 것
- 현재 앱 탭 상태
- 수동 상태 보정
- 경고/확인 필요
- 완료/취소 표시 해석

### 상태 세부 정보가 소유하는 것
- 업무 처리 세부 체크값
- 발송 여부
- 기사 배정 여부
- 계약/반납 진행 여부

### 로그가 소유하는 것
- 누가 언제 무엇을 눌렀는지
- 결과가 어땠는지

### 시트 상태 반영 메타가 소유하는 것
- 어떤 최종 상태 체크가 시트에 반영됐는지
- 반영 성공/실패가 어땠는지

## 6. 물리 설계 시 주의점
1. 원본 feed와 ops 상태를 한 테이블에 섞지 않는다.
2. sync가 ops 상태를 덮어쓰지 않게 한다.
3. 원본 예약 키와 내부 ops 키를 분리한다.
4. 개인정보 필드는 접근 제어를 고려한다.
5. 발송/요청/완료 같은 액션은 로그 우선으로 추적 가능해야 한다.
