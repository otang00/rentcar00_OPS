# rentcar00_OPS Google Sheets → Supabase 매핑표 v1

## 1. 문서 목적
이 문서는 Google Sheets `차량현황` 문서의 `예약` 탭과 `일정` 탭을
앞으로 필요한 ops 앱 구조에 어떻게 연결할지 정리한 1차 매핑표다.

중요:
- 현재는 개념 매핑 단계다.
- 실제 신규 테이블명/컬럼명은 아직 확정하지 않는다.
- 목표는 컬럼 하나하나를 어디서 소유할지 정하는 것이다.

## 3. 소유권 원칙
각 필드는 아래 셋 중 하나로 분류한다.

1. **원본 유지 필드**
   - Google Sheets에서 들어오고, 원본 의미 그대로 유지
2. **기존 Supabase 재사용 필드**
   - 현재 `cars` 등에서 일부 재사용 가능
3. **ops 앱 신규 상태 필드**
   - 원본에 없고 업무처리 앱이 새로 가져야 하는 값

추가 원칙:
- 시트로 되돌려 쓰는 값은 세부 업무 상태 전체가 아니라 최종 최소 상태 체크로 제한한다.

## 4. 원본 컬럼별 매핑표
| 원본 컬럼 | 의미 | 기존 Supabase 대응 후보 | ops 앱 내부 필요 여부 | 비고 |
|---|---|---|---|---|
| 예약ID | 원본 행/예약 고유 식별값 | 현재 직접 대응 없음. 외부 예약 키로 보관 필요 | 예 | 현재 가장 안정적인 원본 키 후보 |
| 예약번호 | 대외/업무용 예약번호 | `booking_orders.public_reservation_code` 와는 별개 개념 가능성 | 예 | 공란 1건 존재, 보조 식별자로 유지 권장 |
| 차량번호 | 예약 대상 차량번호 | `cars.car_number`, `ims_sync_reservations.car_number` | 예 | 차량 매칭 1차 기준 |
| 차종 | 시트 기준 차종명 | `cars.name`, `cars.display_name` 참고 가능 | 예 | 표기 차이 가능성 있어 원본 문자열 보존 필요 |
| 대여일 | 예약 시작 일시 | `ims_sync_reservations.start_at`, `booking_orders.pickup_at` | 예 | Excel datetime → timestamptz 변환 규칙 필요 |
| 반납일 | 예약 종료 일시 | `ims_sync_reservations.end_at`, `booking_orders.return_at` | 예 | Excel datetime → timestamptz 변환 규칙 필요 |
| 배반차위치 | 배차/반납 위치 관련 텍스트 | `ims_sync_reservations.delivery_address`, `pickup_address`, `dropoff_address` 일부 참고 | 예 | 단일 컬럼이므로 앱 내부에서는 구조 분리 필요 |
| 임차인 | 고객명 | `ims_sync_reservations.customer_name`, `booking_orders.customer_name` | 예 | 원본 문자열 그대로 유지 |
| 고객번호 | 고객 전화번호 | `ims_sync_reservations.customer_phone`, `booking_orders.customer_phone` | 예 | 하이픈 유무 정규화 규칙 필요 |
| 생년월일 | 고객 생년월일 | `booking_orders.pricing_snapshot.customerBirth` 일부 유사 | 예 | 개인정보 필드이므로 접근권한 주의 |
| 소개처 | 유입/채널 | `booking_orders.booking_channel` 와 일부 의미 겹칠 수 있으나 동일하지 않을 수 있음 | 예 | 원본 문자열 보존 필요 |
| 결제금액 | 결제 또는 예약 금액 | `booking_orders.quoted_total_amount`, `pricing_snapshot.finalAmount` 참고 가능 | 예 | 금액 의미가 결제완료 금액인지 확인 필요 |
| 예약상태 | 원본 운영 상태 | `ims_sync_reservations.status`, `booking_orders.booking_status` 와 직접 동일하지 않을 수 있음 | 예 | 앱 탭 상태와 분리 관리 |

## 5. 예약 탭 + 일정 탭 결합 해석
### 일정 탭 확인 요약
확인된 일정 탭 컬럼:
- 일정번호
- 예약번호
- 차량번호
- Status
- Date
- 차종
- 위치
- 상세정보
- 가반납
- 예약ID
- 일정완료

핵심 해석:
- `예약ID` 로 예약 탭과 연결 가능한 구조가 보인다.
- `Status` 는 배차 / 반납 / 기타 일정 구분값 역할이다.
- `Date` 는 일정 기준 시각이다.
- `일정완료` 는 일정 처리 여부 플래그 후보다.
- 다만 일부 행은 메모형/레거시 데이터라 정규화가 먼저 필요하다.

## 6. 현재 기존 Supabase 구조와의 관계

- `ims_sync_reservations`
- `booking_orders`

설명:
- 이번 ops 앱은 Google Sheets `예약` + `일정` 탭만 원본 기준으로 사용한다.
- 위 테이블은 참조 자산으로 남아 있을 수 있지만, 매핑의 기준 source 로 사용하지 않는다.

## 8. 앱 탭 상태 계산 규칙과 원본 컬럼의 관계
원본 `예약상태` 만으로는 앱 탭을 만들 수 없다.

| 앱 탭 상태 | 원본에서 가져오는 값 | 추가로 필요한 값 |
|---|---|---|
| 예약중 | 예약상태, 대여일, 반납일 | 준비 미완료 여부 |
| 오늘배차 | 대여일, 예약상태 | 오늘 여부, 고객 인수 완료 여부 |
| 배차중 | 예약상태, 대여일, 반납일 | 고객 인수 완료 여부, 이용 중 상태 |
| 반납일 | 반납일, 예약상태 | 오늘 여부, 반납 완료 여부 |
| 완료/기록 | 예약상태 | 반납 완료 여부 |

핵심:
- 원본 상태는 참고값
- 앱 탭 상태는 계산값

## 9. 권장 키/매핑 전략
### 8-1. 예약 키
권장 방향:
- 원본 예약 키: `예약ID`
- 보조 표시 키: `예약번호`
- 앱 내부 키: 별도 내부 식별값 생성 가능

### 8-2. 차량 매핑 키
권장 방향:
- 1차 매핑: `차량번호` ↔ `cars.car_number`
- 2차 보정: 차종/차량명 비교

### 8-3. 상태 매핑
권장 방향:
- 원본 `예약상태`는 외부 상태 원문 보존
- 앱 내부에는 별도 `업무처리 상태` 계산/저장

### 8-4. 주소 매핑
권장 방향:
- 원본 `배반차위치` 문자열은 원문 그대로 보존
- 앱 내부에서는 필요 시 아래처럼 구조화
  - 배차 주소 원문
  - 반납 주소 원문
  - 대략 주소
  - 위치 메모

## 10. 현재 확인된 리스크
### 9-1. 날짜 공란이 많다
- 대여일/반납일 공란 43건
- 원인 확인 필요
  - 예약취소라 비어 있는지
  - 외부 채널 예약 구조 차이인지
  - AppSheet 계산값인지

### 9-2. 주소가 단일 컬럼이다
- 배차와 반납 주소가 분리되지 않음
- 앱에서는 상태별 업무에 따라 분리 저장이 필요할 가능성이 높다

### 9-3. 예약상태 표현이 앱 목적과 다르다
- 시트 상태는 원장 상태
- 앱 상태는 업무 단계
- 둘을 혼용하면 안 된다

### 9-4. `소개처`의 의미가 불명확하다
- 유입채널인지
- 파트너 식별값인지
- 내부 분류값인지
확인 필요

## 11. 현재 시점 권장 매핑 결론
1. Google Sheets `예약` 탭은 **원본 예약 source** 로 본다.
2. Google Sheets `일정` 탭은 **원본 일정 source** 로 본다.
3. `cars` 는 **차량 마스터** 로 재사용한다.
4. ops 앱은 **새 업무처리 상태 계층**을 별도로 둔다.
5. `예약ID`, `차량번호`, `대여일`, `반납일`, `예약상태` 는 초기 예약 매핑 핵심 축이다.
6. `일정번호`, `예약ID`, `Status`, `Date`, `일정완료` 는 일정 매핑 핵심 축이다.
7. `오늘배차`, `반납일`, `확인 필요` 같은 상태는 **앱 내부 계산 상태**로 본다.
8. 시트로는 최종 최소 상태 체크만 반영한다.
