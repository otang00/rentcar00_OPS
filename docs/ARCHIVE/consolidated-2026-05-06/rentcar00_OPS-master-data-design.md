# rentcar00_OPS Master Data Design

## 1. 목적
이 문서는 원본 시트와 업무 DB의 역할을 나누고,
어떤 데이터가 어디에서 읽히고 어디에 저장되는지 정리한다.

---

## 2. 원본 데이터 기준
원본은 Google Sheets `차량현황` 문서다.
사용 탭은 아래 2개다.
- `예약`
- `일정`

이번 앱 기준에서 아래는 사용하지 않는다.
- `ims_sync_reservations`
- `booking_orders`

---

## 3. 예약 탭 구조
확인된 컬럼:
- 예약ID
- 예약번호
- 차량번호
- 차종
- 대여일
- 반납일
- 배반차위치
- 임차인
- 고객번호
- 생년월일
- 소개처
- 결제금액
- 예약상태

이 탭은 예약 원본이다.

---

## 4. 일정 탭 구조
확인된 컬럼:
- 일정번호
- 예약번호
- 차량번호
- Status
- Date
- 차종
- 위치
- 상세정보
- 가반납
- 예약ID
- 일정완료

이 탭은 일정 원본이다.

주의:
- 현재 일정 탭에는 예약 연결형 행과 레거시 메모형 행이 섞여 있다.
- 이 문서에서는 존재만 확정하고,
  정규화 규칙은 추후 별도 확정한다.

---

## 5. 연결 기준
우선 연결 기준은 아래 순서로 본다.
1. `예약ID`
2. `예약번호`
3. `차량번호`

원칙:
- 예약 기본 연결키는 `예약ID` 우선이다.
- 일정은 가능하면 `예약ID`로 예약과 연결한다.
- 보조 연결이 필요할 때만 `예약번호`, `차량번호`를 본다.

---

## 6. 역할 분리
### 6-1. Google Sheets
역할:
- 예약/일정 원본 보존
- 기존 현황판 기준 유지
- 최종 상태 체크 최소 반영

### 6-2. Supabase
역할:
- 업무 상태 저장
- 버튼 실행 결과 저장
- 로그 저장
- 메모/예외 처리 저장
- 시트 반영 기록 저장

즉,
시트는 원본,
Supabase는 업무 처리 저장소다.

---

## 7. 데이터 흐름 원칙
1. 시트 `예약` / `일정`을 읽는다.
2. 앱은 Supabase의 업무 상태/로그와 결합해서 탭 리스트를 만든다.
3. 사용자가 카드에서 버튼을 누르면 Supabase 상태가 바뀐다.
4. 최종 완료 시 필요한 상태 체크만 시트에 반영한다.

---

## 8. 이번 문서에서 확정된 것
- 원본 시트는 `예약` + `일정`
- 업무 상태와 로그는 Supabase 관리
- 최종 시트 반영은 최소 상태 체크만 수행
- 기존 `ims_sync_reservations`, `booking_orders` 는 제외

---

## 9. 아직 확정하지 않은 것
- 일정 탭 레거시 행 정규화 규칙
- 시트에 반영할 정확한 상태 체크 항목
- 업무 상태의 최종 명칭 세트
- 실제 DB 테이블/컬럼 이름
- sync/outbound 실행 주기와 방식

위 항목은 아직 확정하지 않는다.
