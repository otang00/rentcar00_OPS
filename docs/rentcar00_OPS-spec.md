# rentcar00_OPS Spec

## 1. 문서 목적
이 문서는 `rentcar00_OPS` 모바일 업무처리 앱의 제품 목표와 MVP 범위를 고정하기 위한 기준 문서다.

이 앱은 기존 AppSheet 현황판을 대체하지 않는다.
기존 AppSheet는 차량/예약 현황 확인용으로 유지하고,
새 Flutter 앱은 예약 상태별 업무를 실제로 처리하는 전용 앱으로 설계한다.

## 2. 제품 정의
- 플랫폼: Flutter 기반 Android 우선 모바일 앱
- 데이터 백엔드: Supabase
- 원천 데이터: 기존 AppSheet가 연결된 Google Sheets `차량현황` 문서의 `예약` 탭 + `일정` 탭
- 1차 목표: 예약 1건이 들어온 뒤부터 반납 완료까지의 업무 진행을 상태별로 처리

## 3. 역할 분리
### 기존 AppSheet
- 차량/예약 현황판
- 일정 확인
- 전체 현황 조회

### 새 Flutter 앱
- 상태별 업무 처리
- 업무 버튼 실행
- 문자/연락처/탁송요청문 생성
- 상태 전이
- 처리 로그 저장
- 업무 상태는 Supabase에서 관리
- 최종적으로 필요한 최소 상태 체크만 Google Sheets에 반영

핵심 원칙:
이 앱은 “보는 앱”이 아니라 “처리하는 앱”이다.

## 4. 업무 대상 흐름
예약은 아래 흐름을 중심으로 처리한다.

`예약중 → 오늘배차예약 → 배차중 → 반납일 예약건 → 반납완료`

중요:
- 원본 기준 상태는 Google Sheets `예약` 탭의 `예약상태` 컬럼이다.
- 원본 상태값은 아래 4개로 고정한다.
  - `예약중`
  - `예약취소`
  - `배차중`
  - `반납완료`
- 앱은 이 원본 상태값을 버리지 않고 유지한다.
- 앱 내부에서는 별도 복잡한 상태값보다 **탭 표시용 `tab_key` 계산**을 우선한다.

탭 계산 기준 요약:
- 예약중 탭: `예약상태='예약중'` 이면서 `대여일`이 오늘이 아닌 건
- 오늘배차 탭: `예약상태='예약중'` 이면서 `대여일`이 오늘인 건
- 배차중 탭: `예약상태='배차중'` 이면서 `반납일`이 오늘이 아닌 건
- 반납일 탭: `예약상태='배차중'` 이면서 `반납일`이 오늘인 건
- 완료 탭: `예약상태='반납완료'`
- `예약취소` 는 기본 탭에서 제외한다.

## 5. 핵심 상태 정의
### 5-1. 예약중
- 예약은 되었으나 배차 전 준비가 필요한 상태
- 핵심 업무: 연락처 저장, 신분증/주소 요청, 답변 확인

### 5-2. 오늘배차예약
- 오늘 차량이 고객에게 배차될 예정인 상태
- 핵심 업무: 고객 배차 안내, 탁송 요청, 기사 배정, 기사 안내, 계약서 확인/발송

### 5-3. 배차중
- 차량이 고객에게 인도되었거나 이용 중인 상태
- 핵심 업무: 차량 위치/상태 확인, 이용 중 안내, 반납 전 안내, 연장 확인

### 5-4. 반납일 예약건
- 오늘 반납 예정인 상태
- 핵심 업무: 반납 안내, 회수 탁송 요청, 기사 안내, 반납 완료 처리

### 5-5. 반납완료
- 반납이 완료된 상태
- 기본 업무 탭에서는 제외하고 완료 기록 또는 검색으로만 조회

## 6. 탭 구조
앱의 기본 탭 구성은 아래 5개로 고정한다.

1. 예약중
2. 오늘배차
3. 배차중
4. 반납일
5. 완료

각 탭에는 해당 상태의 예약 카드 리스트가 표시된다.

탭 키는 아래로 고정한다.
- 예약중 → `rc00_ops_tab_pending`
- 오늘배차 → `rc00_ops_tab_pickup_today`
- 배차중 → `rc00_ops_tab_in_use`
- 반납일 → `rc00_ops_tab_return_due`
- 완료 → `rc00_ops_tab_completed`

탭 상태는 시트 `예약상태` 원문 + 날짜 기준으로 계산한다.
원본 `예약상태` 값은 별도 `status_raw` 로 유지한다.

## 7. 카드 표시 정보
카드는 **한 줄 중심의 얇은 리스트 카드**를 기본으로 한다.

공통 최소 필드:
- 고객명
- 차량번호
- 일정 기준 시각 1개
- 위치 요약
- 핵심 경고 배지 1~2개

탭별 기준 시각:
- 예약중: `start_at`
- 오늘배차: `start_at`
- 배차중: `end_at`
- 반납일: `end_at`
- 완료: `completed_at` 이 있으면 우선, 없으면 `end_at`

탭별 핵심 배지:
- 예약중: 신분증 미확보 / 주소 미확보
- 오늘배차: 준비 미완료 / 탁송·계약·서명 미완료
- 배차중: 반납 임박 / 연장·이슈
- 반납일: 반납완료 직전 미처리
- 완료: 특이사항

원칙:
- 연락처 일부, 차량명, 세부 상태 배지는 상세 진입 전 꼭 필요할 때만 보조 노출한다.
- 카드에는 현재 탭에서 중요한 경고만 노출하고, 전체 정보는 상세에서 본다.
- 카드 선택 시 예약 상세 화면으로 이동한다.

## 8. 예약 상세 화면
예약 상세 화면에는 아래 내용을 포함한다.
- 고객 정보
- 예약 정보
- 차량 정보
- 배차/반납 주소
- 현재 업무 상태
- 상태별 업무 버튼
- 업무 처리 로그
- 메모

화면 배치 원칙:
- 탭 네비게이션 아래에 현재 상태 요약을 둔다.
- 상세 상단에는 상태/배지 영역을 둔다.
- 액션 버튼 영역은 상태 요약 아래, 상세 본문 위에 둔다.
- 완료 판정 체크와 로그는 액션 버튼 아래에 둔다.

핵심 구분:
- 액션 버튼은 `rc00_ops_action_*` 실행 단위다.
- 완료 체크는 `rc00_ops_check_*` 판정 단위다.
- MVP 기준 탭 전이는 복잡한 내부 `status_key` 보다 시트 `예약상태` 원문과 날짜 계산을 우선한다.

## 11. 자동 탭 이동 규칙
탭 이동 source of truth 는 상태 레코드의 `tab_key` 다.

고정 전이 기준:

### 예약중 → 오늘배차
- `start_at` 날짜가 오늘이면 이동한다.
- 준비 미완료여도 오늘배차로 이동한다.
- 대신 준비 미완료 카드는 주황 경고로 표시한다.

### 오늘배차 → 배차중
- `rc00_ops_action_request_delivery` 실행만으로는 이동하지 않는다.
- **실제 출발 확인값**이 잡히면 배차중으로 이동한다.
- MVP에서는 `rc00_ops_status_dispatch_in_progress` 또는 수동 확인을 실제 출발 확인값으로 본다.

### 배차중 → 반납일
- `end_at` 날짜가 오늘이 되는 **00:00** 에 반납일로 이동한다.
- 반납 하루 전은 배차중에 유지한다.
- 대신 노랑 경고로 반납 임박을 표시한다.

### 반납일 → 완료
- `rc00_ops_action_complete_return` 완료 시 이동한다.

### 완료 → 기본 탭 제외
- 반납완료 후 7일간 완료 탭에 노출한다.
- 7일이 지나면 완료 탭 기본 목록에서는 제외하고 검색으로만 조회한다.

예외 원칙:
- 긴급/확인필요는 어느 탭이든 배지 표시
- 날짜 기준과 수동 상태가 충돌하면 수동 상태 우선 + 경고 표시

MVP 전이 해석 고정:
- `request_delivery` 는 요청 실행일 뿐, 배차중 전이 트리거가 아니다.
- 실제 출발 확인은 별도 확인 액션/체크로 남긴다.
- `complete_return` 만 완료 전이의 강한 트리거로 본다.

## 12. 탭별 완료/경고 규칙
### 예약중
- `신분증 확보`, `주소 확보`, `배차준비완료` 체크가 핵심이다.
- 준비 미완료 카드는 기본 경고 배지를 유지한다.

### 오늘배차
- 준비 미완료 상태면 카드 강조색은 주황으로 둔다.
- `배차안내`, `탁송요청`, `계약/서명 관련 체크`가 핵심이다.

### 배차중
- 반납일이 내일인 건은 카드 강조색을 노랑으로 둔다.
- `서명확인`, `반납 전 안내`, `연장 확인`, `긴급/사고 대응`이 핵심이다.

### 반납일
- `반납안내`, `탁송요청`, `반납지 변경`, `추가요금 안내`, `반납완료`가 핵심이다.

### 공통
- 액션 버튼은 실행되면 `idle → done` 등 상태색이 바뀐다.
- 액션 실행과 완료 판정은 분리한다.
- 체크가 완료되지 않으면 다음 탭으로의 자동 이동을 막거나 `확인 필요`로 남긴다.

## 13. 문자 템플릿 요구사항
필요 템플릿:
1. 신분증/주소 요청 문자
2. 예약 확정 또는 준비 안내 문자
3. 고객 배차 안내 문자
4. 차량 출발 안내 문자
5. 탁송기사 안내 문자
6. 반납 전날 안내/연장 확인 문자
7. 반납일 안내 문자
8. 반납 탁송기사 안내 문자

원칙:
- 템플릿은 하드코딩하지 않고 DB 관리 가능 구조로 설계
- 예약 데이터 치환값 삽입 가능해야 함
- MVP는 문자앱을 열고 본문만 미리 채우는 방식
- 추후 Solapi 등 API 연동 가능하게 확장

## 14. 외부 연동 후보
- Google Sheets `차량현황` 문서 (`예약` / `일정` 탭)
- IMS 계약서
- IMS Connect
- 탁송 요청 시스템
- 문자 발송 서비스
- 시동제어 API

원칙:
- 예약 원본과 일정 원본은 Google Sheets 기준으로 본다
- `ims_sync_reservations`, `booking_orders` 는 이번 ops 앱의 기준 원장으로 사용하지 않는다
- Supabase는 `예약 원장 / 예약별 상태 / 액션 로그` 계층으로 분리한다
- 예약별 상태는 예약 1건당 1행을 기본으로 하며 `tab_key`, `status`, 현재 체크값을 가진다
- 시트는 원본 유지가 우선이며, 최종적으로 필요한 최소 상태 체크만 반영하는 방향으로 본다
- OPS 앱은 AppSheet API를 직접 호출하지 않는다.
- OPS 앱의 외부 반영 신호는 Google Sheets 수정으로만 보낸다.
- 기존 AppSheet/봇은 시트 변경에 반응하는 후속 자동화 레이어로 본다.
- MVP에서는 수동 체크, 외부 링크, 요청문 생성, mock 상태 변경 위주로 처리

## 15. Google Sheets 반영 / 트리거 원칙
### 15-1. 구조 원칙
- 입력 원장: Google Sheets `예약` / `일정`
- 내부 처리 소스: Supabase
- 외부 반영 신호: Google Sheets 수정
- AppSheet: 직접 제어 대상이 아니라 시트 변경 반응자

### 15-1A. MVP 최소 outbound 고정
- outbox 생성 대상은 4개만 고정한다.
  - `request_delivery`
  - `change_end_at`
  - `change_dropoff_address`
  - `complete_return`
- 나머지 액션은 내부 로그 + 내부 체크만 처리한다.
- 실제 시트 apply는 마지막 phase 별도 승인 전까지 금지한다.

### 15-2. OPS 앱이 직접 책임지는 것
- 업무 상태 저장
- 실행 로그 저장
- 카드/탭 계산
- 마지막 phase에서만 승인된 최소 시트 컬럼 수정

### 15-3. OPS 앱이 직접 하지 않는 것
- AppSheet 내부 계산식 제어
- AppSheet Action 직접 호출
- AppSheet 내부 Bot/Automation 제어

### 15-4. 트리거 명세 작성 원칙
각 액션은 나중에 아래 형태로 잠근다.
1. 어떤 액션이 실행되는지
2. 어떤 시트의 어떤 컬럼을 수정하는지
3. 그 수정이 어떤 후속 반응을 유발하는지
4. 실패 시 앱 내부 fallback이 무엇인지

현재 단계 결론:
- AppSheet 계산 로직 자체를 먼저 복제하는 것이 아니라
- Google Sheets 수정 트리거를 정확히 정의하는 방향으로 간다.
- 단, 실제 Google Sheets write는 최종 승인 전까지 금지한다.
- 개발/설계/초기 검증 단계에서는 read-only + dry-run만 허용한다.

## 17. 구현 전 원칙
- 기존 Google Sheets 구조 확인 전까지 실제 컬럼명/테이블명/함수명을 임의 확정하지 않는다.
- 개념 모델 중심으로 먼저 설계한다.
- 기존 AppSheet 현황판과 역할이 겹치지 않게 유지한다.
- 기준 원장은 Google Sheets `차량현황`의 `예약` + `일정` 탭이다.
- 업무 세부 상태와 로그는 Supabase가 소유한다.
- 시트에는 최종적으로 필요한 최소 상태 체크만 반영하는 방향으로 설계한다.
- 상태별 탭 이동, 업무 버튼 실행, 로그 저장을 최우선 가치로 둔다.
- AppSheet 내부 계산 로직보다 Google Sheets 수정 트리거를 우선 명세화한다.
- AppSheet virtual column / 계산 로직은 미확인이더라도, 시트 수정 기반 후속 반응 구조는 기준으로 고정한다.
- Google Sheets write는 실운영 트리거로 간주하고, 최종 phase 전까지 금지한다.
- 초기 구현 단계에서는 outbound write를 수행하지 않고 dry-run payload / outbox 적재까지만 허용한다.
- 일정 정규화는 `reservation_id` 우선, 그다음 `reservation_number` unique match, 실패 시 orphan raw 유지로 고정한다.
