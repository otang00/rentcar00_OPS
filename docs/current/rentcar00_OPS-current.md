# rentcar00_OPS Current

## 문서 역할
이 문서는 rentcar00_OPS의 유일한 현재 active 실행 문서다.
현재 실행 중인 작업 1건만 적는다.
완료된 기능과 운영 확인 포인트는 `docs/completed/rentcar00_OPS-completed.md`로 옮긴다.

---

## 현재 active 작업
**관리자 기능 준비 + 기능 버튼 UI 정리 + 일정상세 헤더 보정**

## 목적
관리자만 접근 가능한 운영 메뉴를 만들고, 기능 버튼이 계속 늘어나는 상황에 맞춰 버튼 UI 기준을 먼저 정리한다. 동시에 일정상세 화면에서 차량번호 가독성을 높인다.

---

## 1. 관리자 기능 범위
### 진입
- 좌상단 `빵빵카` 텍스트/영역을 눌러 관리자 메뉴로 진입한다.
- 관리자 권한(`role == admin`)만 접근 가능하다.
- 일반 직원은 접근 차단 안내를 본다.

### 1차 관리자 메뉴
1. 직원관리
   - 직원 목록
   - 직원 추가
   - 권한 변경: `admin / staff`
   - 활성/비활성 처리
   - 로그인 ID / 표시명 관리
2. 차량관리
   - 차량 추가
   - 차량 정보 변경
   - 차량 삭제 또는 비활성 처리
3. 작업로그
   - 누가
   - 언제
   - 어떤 작업을 했는지
   - 관련 예약/차량/직원 기준으로 조회
4. 출근확인
   - Wi‑Fi 기반 출근 버튼
   - 직원별 출근 로그
   - 관리자 출근현황
5. 앱푸시
   - 직원 기기 push token 저장
   - 관리자 공지/알림 발송
   - 자동 알림은 후속 phase

### 관리자 기능 리스크
- 직원 추가는 Supabase Auth 계정 생성과 `rc00_ops_staff_accounts` row 생성이 함께 필요하다.
- 차량 삭제는 과거 예약/로그와 연결될 수 있으므로 실제 삭제보다 비활성 처리를 우선 검토한다.
- 작업로그는 기존 예약 action log와 관리자/차량/직원 변경 로그를 같은 화면에서 볼지, 별도 테이블로 분리할지 결정이 필요하다.
- 앱푸시는 FCM 설정/권한/토큰 저장/서버 발송 경로가 필요해 관리자 MVP 이후 별도 phase가 적합하다.
- Wi‑Fi 출근확인은 Android/iOS 권한 차이가 있고 SSID 위조 가능성이 있어 보조 확인 수단으로 본다.

## 2. 기능 버튼 UI 기준
### 현재 상태
- 예약상세 기능 카드는 2열 그리드 카드형이다.
- 현재 버튼 예:
  - `수정`
  - `차량변경`
  - `배차완료`
  - `반납완료`
  - `전화`
  - `문자`
  - `IMS추가` / `IMS등록됨`
- `수정`, `배차완료`, `반납완료`, `IMS추가`가 모두 강조색 계열이라 중요도가 섞인다.
- 상태보드 즉시배차/세차는 리스트 카드형 다이얼로그를 사용한다.
- 반납완료는 OPS 상태뿐 아니라 IMS active binding까지 건드릴 수 있어 일반 보조 버튼과 같은 레벨로 두면 실수 가능성이 있다.

### 잠긴 UI 기준
#### A. 주요 lifecycle 영역
- 예약상세 상단에 별도 영역으로 둔다.
- 상태에 따라 하나만 표시한다.
  - 미배차: `배차완료`
  - 배차중: `반납완료`
- 한 줄 큰 버튼으로 표시한다.
- 실행 전 확인 다이얼로그를 반드시 띄운다.
- 색상/아이콘은 다른 보조 버튼보다 강하게 두되, `수정`/`IMS추가`와 같은 강조색을 공유하지 않는다.

#### B. 관리 액션 영역
- `수정`, `차량변경`, `IMS추가`/`IMS등록됨`을 묶는다.
- 작은 카드형 버튼으로 유지한다.
- 관리 액션은 상태 변경 lifecycle 버튼보다 시각 우선순위를 낮춘다.

#### C. 연락 액션 영역
- `전화`, `문자`를 묶는다.
- 작은 카드형 버튼 또는 톤다운 버튼으로 둔다.
- lifecycle/관리 액션과 분리해 실수 클릭 가능성을 낮춘다.

#### D. 다이얼로그 일관성
- 즉시배차/세차처럼 선택지가 있는 기능은 리스트 카드형 다이얼로그를 유지한다.
- 완료/취소성 action은 확인 다이얼로그를 사용한다.
- 다이얼로그 문구는 `무엇이 바뀌는지`와 `IMS 동시 처리 여부`를 같이 보여준다.

## 3. 일정상세 헤더 보정
### 요구사항
- 일정상세 화면 맨 위에서 날짜 밑 차량번호를 더 크게 보이게 한다.
- 차량번호는 날짜보다 살짝 크게 표시한다.

### 현재 구조
- `StatusBoardDetailPage`에서 일정 row는 `_ScheduleDetailBody`로 표시한다.
- 현재 헤더 순서:
  1. 일정 타입
  2. 날짜/시간
  3. 차량번호 badge
- 현재 차량번호는 `titleSmall` 수준이라 날짜보다 작다.

### 기준
- 날짜/시간 아래 차량번호 badge는 유지한다.
- 차량번호 텍스트는 날짜보다 살짝 큰 크기 또는 `titleLarge/headlineSmall` 수준으로 올린다.
- badge padding도 조금 키워 모바일에서 탭하지 않아도 바로 보이게 한다.

## Phase 계획
### Phase 0 — 실행 전 UI 체크
종료 조건:
- 사장님이 기능 버튼 UI 기준과 관리자 phase 순서를 확인한다.
- 구현 범위를 `Phase 1만` 또는 `전체 진행` 중 하나로 확정한다.

### Phase 1 — 일정상세 헤더 보정
종료 조건:
- 일정상세 날짜 밑 차량번호가 날짜보다 살짝 크게 보인다.
- 기능 변경 없이 UI만 보정한다.

### Phase 2 — 예약상세 기능 버튼 UI 정리
종료 조건:
- lifecycle / 관리 / 연락 영역이 분리된다.
- 배차완료/반납완료는 한 줄 큰 버튼으로 표시된다.
- 확인 다이얼로그 문구가 실제 변경 내용과 IMS 동시 처리 여부를 명확히 보여준다.

### Phase 3 — 관리자 진입/권한 뼈대
종료 조건:
- 좌상단 `빵빵카`에서 관리자 메뉴 진입 가능하다.
- `role == admin`만 진입 가능하다.
- 일반 직원은 차단 안내를 본다.
- 관리자 메뉴에는 직원관리/차량관리/작업로그/출근확인/앱푸시 placeholder가 보인다.

### Phase 4 — 직원관리 MVP
종료 조건:
- 직원 목록을 볼 수 있다.
- 직원 추가 flow 기준이 정리된다.
- 권한/활성 상태 변경 방식이 확정된다.

### Phase 5 — 차량관리 MVP
종료 조건:
- 차량 목록을 볼 수 있다.
- 차량 추가/수정/비활성 처리 기준이 확정된다.
- 과거 예약과 연결된 차량 삭제는 차단 또는 비활성 우선으로 처리한다.

### Phase 6 — 작업로그 화면
종료 조건:
- 기존 action log 조회 기준을 확인한다.
- 관리자/차량/직원 변경 로그 저장 구조를 확정한다.
- 관리자 화면에서 최근 작업로그를 볼 수 있다.

### Phase 7 — 출근확인/앱푸시 설계
종료 조건:
- Wi‑Fi 출근확인에 필요한 권한/패키지/DB 구조를 확정한다.
- FCM 기반 앱푸시 도입 여부와 서버 발송 경로를 확정한다.
- 구현은 별도 승인 후 진행한다.
