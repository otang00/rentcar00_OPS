# rentcar00_OPS Sync 전략 v1

## 1. 목적
Google Sheets `차량현황` 문서의 `예약` 탭과 `일정` 탭 원본 데이터를
기존 Supabase 프로젝트 안으로 가져오되,
원본과 ops 앱 상태를 섞지 않고 운영하기 위한 1차 sync 전략이다.

## 2. 원칙
1. 원본을 덮어쓰지 않는다.
2. sync는 원본 데이터 갱신만 담당한다.
3. 앱 업무 상태는 별도 계층에서 관리한다.
4. 동일 예약 식별과 차량 연결 규칙을 먼저 잠근다.
5. 시트에는 최종 최소 상태 체크만 outbound 반영한다.

## 3. 권장 흐름
### 1단계
Google Sheets `예약` 탭 + `일정` 탭 읽기

### 2단계
원본 row를 표준화
- 날짜 변환
- 전화번호 정규화
- 공백/문자열 정리

### 3단계
차량번호로 차량 마스터 연결 시도
- 1차: `cars.car_number`
- 실패 시 보정/검토 대상으로 분리

### 4단계
원본 예약 feed + 원본 일정 feed 계층 upsert
- 예약은 원본 예약키 기준
- 일정은 일정키 + 예약 연결키 기준
- 원본 상태/일정 종류 원문 보존

### 5단계
ops projection 갱신
- 예약/일정 결합
- 앱 탭 상태 계산
- 주요 경고 플래그 계산
- 기존 업무 상태는 보존

### 6단계
시트 상태 체크 outbound 반영
- 최종 완료 또는 반영이 필요한 최소 상태만 갱신
- 세부 로그/이력은 시트로 쓰지 않음
- 반영 성공/실패는 별도 기록

## 4. 절대 하지 말 것
- sync가 업무 상태를 초기화하는 것
- sync가 수동 메모를 덮어쓰는 것
- sync가 액션 로그를 삭제하는 것
- 시트로 세부 업무 로그를 되밀어 넣는 것
- 시트로 앱 내부 세부 상태 전부를 쓰는 것
- IMS read model을 원장처럼 덮어쓰는 것

## 5. 충돌 처리 원칙
### 원본 변경 vs 앱 상태
- 원본 일정/고객/차량 정보는 갱신 가능
- 앱 내부 발송/체크/로그 상태는 유지

### 날짜 충돌
- 대여/반납일 변경 시 탭 재분류는 가능
- 단, 수동 상태 보정이 있으면 경고 표시

### 차량 매칭 실패
- 앱에서 `확인 필요`로 표시
- 자동 진행 금지

## 6. MVP 수준 sync 방식
1차 구현 권장:
- 수동 실행 또는 간단한 admin-trigger sync
- 매번 전체 스냅샷 또는 변경분 비교
- 결과 요약 로그 저장
- 시트 outbound 반영은 최소 상태 체크만 수행

## 7. 다음 확정 필요 사항
- AppSheet가 실제로 어떤 row key를 쓰는지
- Google Sheets 변경 감지 방식
- 일정 탭의 레거시 메모형 행 처리 방식
- 취소/완료 예약 보존 기간
- 같은 예약번호 중복 가능성 여부
